The Real-World Problem

Disaster response edge nodes run dangerous mixtures of workloads

A single field server at a wildfire perimeter simultaneously runs AI drone analysis, responder coordination APIs, sensor fusion, and background logging. These workloads destroy each other's performance under naive scheduling.

What happens under Linux CFS:
An LLM inference burst consumes 28 GB/s memory bandwidth and 42% LLC miss rate. The dispatch API's LLC miss rate rises from 6% to 18%, adding 22ms overhead per request. P99 latency spikes from 145ms to 12,738ms. Responders lose situational awareness at the worst possible moment.
Why Existing Schedulers Fail

Every standard scheduler has a fundamental blind spot

Linux CFS
Optimises fairness — no concept of deadlines, SLA requirements, or cache interference. Treats drone AI and dispatch API as equals.
Round Robin
Preempts GPU kernels mid-inference. Each context switch adds 2–10ms overhead. Ruins inference throughput.
Static Priority
Priority assigned at submission never changes. A task that was low-priority 30s ago may be critically urgent now as its deadline approaches.
Kubernetes Default
Understands CPU/memory requests. Completely blind to LLC cache interference — the actual cause of tail latency.
The Core Gap
None of these schedulers model the interference relationship between workloads — how much one process degrades another through shared LLC and memory bus contention.
None have deadline-aware urgency that dynamically reprioritises as deadlines approach.
None integrate energy feedback from hardware power counters into the scheduling decision loop.
MOSAIC's Innovation

Four novel mechanisms that don't exist together in any open-source scheduler

1

Interference Matrix

A 6×6 pairwise matrix measuring exactly how much each workload class degrades every other through LLC cache pollution. Built via perf_event_open() hardware counters. If ipc_degradation > 35%, the pair is never colocated.

2

Dynamic Urgency Score

Recomputed every 100ms: U(t) = tier_weight / (remaining_ms / deadline_ms). As deadline approaches, urgency → ∞. Past-deadline tasks get U=∞ and jump to queue head. Eliminates static priority inversion.

3

ML Workload Classifier

Nearest-centroid classifier on a 4D hardware fingerprint: (IPC, LLC miss rate, MemBW, branch miss). With online EWC learning — centroid updates on every completion. Accuracy improves automatically. No training data needed.

4

RAPL Energy Feedback

Reads Intel RAPL power counters every 500ms. When power exceeds cap, throttles lowest-urgency task via cgroups v2 cpu.weight halving (software DVFS). Critical for battery-backed edge nodes.

Interview Questions to Expect
Q: Why userspace, not a kernel module?
Safety + debuggability. Kernel panics take down the machine. Scheduling decisions are millisecond-scale, so userspace TCP round-trips add negligible overhead.
Q: What is Jain's Fairness Index?
JFI = (Σxᵢ)² / (n·Σxᵢ²). Range [1/n, 1]. 1.0 = all workload classes meet deadlines at identical rates — no class is systematically disadvantaged.
Q: How does MOSAIC extend to multi-node?
Add a global admission controller backed by etcd, maintaining per-node interference state. Route tasks to least-interference node. Direct path to Google Borg architecture.
Q: What's NUMA and why does it matter?
Non-Uniform Memory Access — remote NUMA node adds 40–100ns latency vs 4ns local. MOSAIC's cpuset pinning keeps workloads on local NUMA nodes.
